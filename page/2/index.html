redis常见数据类型

常见的5种：

  • String
    • key-value结构，value不仅是字符串也可以是数字，value最长数据长度512M。字符串对象编码方式：int、raw和 embstr。
    • 底层数据结构：int和SDS（简单动态字符串）

      • SDS可以保存文本、二进制数据；获取字符串长度事件复杂度为O(1)，因为用len属性记录了长度。

  • Hash
    • 键值对（key - value）集合，其value是键值对（或键值对列表）
    • 底层数据结构：压缩列表(哈希类型元素个数小于512个且所有值小于64字节)或哈希表———在7.0之后压缩列表废弃，有listpack实现
  • List
    • 字符串列表，按照插入顺序排序，头部、尾部都可以向List列表添加元素，2^32-1最大长度。
    • 底层数据结构：双向链表或压缩列表(列表的元素个数小于512个且值小于64字节)———但是3.2后只有quiklist实现，代替了双向链表和压缩列表。
  • Set
    • 底层数据结构：哈希表或整数集合（元素<512时）。
  • Zset(Sorted Set)
    • 比于 Set 类型多了一个排序属性 score（分值）
    • 底层数据结构：由压缩列表(个数<128且值<64字节)或跳表实现——— Redis 7.0中listpack代替压缩列表。

MySQL基础01

发表于 2024-06-09 更新于 2024-06-12 分类于 八股 阅读次数： Valine：

MySQL 主要分为 Server 层和引擎层，Server 层主要包括连接器、查询缓存、分析器、优化器、执行器，同时还有一个日志模块（binlog），这个日志模块所有执行引擎都可以共用，redolog 只有 InnoDB 有。

引擎层是插件式的，目前主要包括，MyISAM,InnoDB,Memory 等。

  • 查询语句的执行流程如下：权限校验（如果命中缓存）—->查询缓存—->分析器—->优化器—->权限校验—->执行器—->引擎

  • 更新语句执行流程如下：分析器——>权限校验——>执行器—->引擎—-redo log(prepare 状态)—->binlog—->redo log(commit 状态)四 参考

MySQL存储引擎小记

在MySQL中，存储引擎是采用插件式架构，支持多种存储引擎，甚至可以为不同数据表设置不同的存储引擎。存储引擎是基于表的，不是基于数据库的。

常用的四种：MyISAM(5.5之前的默认)、InnoDB(5.5之后默认)、MEMORY、ARCHIVE

MYISAM和InnoDB的区别:

  • InnoDB支持事务，默认使用可重复读的隔离级别（可以解决幻读、不可重复读问题），基于MVCC和Next-Key Lock实现。

引用和对象

  1. new运算符，new创建对象实例（对象实例在堆内存中），对象引用指向对象实例（对象引用存放在栈内存中）。

对象相等和引用相等的区别：
对象相等一般比较的是内存中存放的内容是否相等。
引用相等一般比较的是他们指向的内存地址是否相等。

LRU缓存

hot100—146

请你设计并实现一个满足 LRU (最近最少使用) 缓存 约束的数据结构。
实现 LRUCache 类：LRUCache(int capacity) 以正整数作为容量 capacity 初始化 LRU 缓存

  • int get(int key) 如果关键字 key 存在于缓存中，则返回关键字的值，否则返回 -1 。
  • void put(int key, int value) 如果关键字 key 已经存在，则变更其数据值 value ；如果不存在，则向缓存中插入该组 key-value 。
  • 如果插入操作导致关键字数量超过 capacity ，则应该逐出最久未使用的关键字。
  • 函数 get 和 put 必须以 O(1) 的平均时间复杂度运行。

话不多说，直接上代码。需要注意的一点是，基础操作是增加节点和删除节点，所以优先把增删写了(并且增加节点的操作只会在头节点进行)，后面的就可以直接调用。并且，需要先自己定义双向链表节点。
