集合，也叫做容器

主要由两大接口派生而来：

  • Collection：存单一元素
  • Map：存键值对

ArrayList

继承于AbstractList， 实现了List、RandomAccess(快速随机访问)、Cloneable、java.io.Serializable这些接口。

可以通过元素的序号快速获取元素对象，这就是快速随机访问

ArrayList是 List 的主要实现类，底层使用 Object[]存储，适用于频繁的查找工作，线程不安全。Vector 是 List 的古老实现类，底层使用Object[] 存储，线程安全

扩容机制(JDK8)：以无参数构造方法创建ArrayList时，实际上初始化赋值的是一个空数组。当真正对数组进行添加元素操作时，才真正分配容量。即向数组中添加第一个元素时，数组容量扩为 10。之后用grow方法进行扩容，核心代码int newCapacity = oldCapacity + (oldCapacity >> 1)，所以ArraList每次扩容之后容量都会变为原来的1.5倍左右（使用了位运算）。

JDK6 new 无参构造的 ArrayList 对象时，直接创建了长度是 10 的 Object[] 数组 elementData

插入一个：Array.copyOf和System.arraycopy()方法：
联系:看两者源代码可以发现 copyOf()内部实际调用了 System.arraycopy() 方法
区别:arraycopy() 需要目标数组，将原数组拷贝到你自己定义的数组里或者原数组，而且可以选择拷贝的起点和长度以及放入新数组中的位置。 copyOf() 是系统自动在内部新建一个数组，调用arraycopy()进行复制后返回该数组。

LinkedList

LinkedList基于双向链表实现的集合类。内存地址不连续，只能通过指针来定位。

和ArrayList增删元素的时间复杂度记得要分首尾和指定位置讨论。

LinkedList中返回指定下标元素时候会比较下标值与链表size的一半大小来确定是从链表头还是尾开始遍历，充分利用双向链表的特性。

LinkedList和ArrayList的区别，可以从以下几点分析：

  • 底层实现的数据结构，不同导致的某些行为也不同。
  • 增、删时候复杂度
  • 内存空间占用
  • 相同点：二者都不是线程安全的

Set

HashSet、TreeSet、LinkedListSet的区别

  • 相同点：都不是线程安全的
  • 底层数据结构：HashSet(基于HashMap实现)、TreeSet(基于红黑树，元素有序)、LinkedListSet(基于链表和哈希表，FIFO)
  • 不需要元素插入和取出顺序、需要定制有序场景、需要元素按照FIFO顺序

Queue

Queue扩展了Collection接口，根据因为容量问题而导致的插入、删除失败后处理方式的不同，可以将操作函数分为两种。

Queue接口 抛出异常 返回特殊值
插入队尾 add(E e) offer(E e)
删除队首 remove() poll()
查询队首 element() peek()

而Deque是双端队列，在操作上会区分在队首还是队尾。

Queue接口 抛出异常 返回特殊值
插入队首 addFirst(E e) offerFirst(E e)
插入队尾 addLast(E e) offerLast(E e)
删除队首 removeFirst() pollFirst()
删除队首 removeLast() pollLast()
查询队首 getFirst() peekFirst()
查询队尾 getLast() peekLast()

Deque 还提供有 push() 和 pop() 等其他方法，可用于模拟栈。

ArrayDeque与LinkedList区别：
ArrayDeque 插入时可能存在扩容过程, 不过均摊后的插入操作依然为 O(1)。虽然 LinkedList 不需要扩容，但是每次插入数据时均需要申请新的堆空间，均摊性能相比更慢。从性能的角度上，选用 ArrayDeque 来实现队列要比 LinkedList 更好。此外，ArrayDeque 也可以用于实现栈。

ArrayDeque是基于可变长的数组（是循环数组）和双指针来实现（头尾指针），初始化默认容量为16，当数组满了时候需扩容，扩容时，新数组的容量是旧数组容量的两倍，并重新排列元素。

Map

HashMap和Hashtable区别：

  • 线程安全与不安全（但是需要线程安全的话还是建议使用ConcurrentHashMap而不是Hashtable）
  • 效率上，HashMap更合适
  • 对NullKey的支持（Hashtable不支持）
  • 底层数据结构（HashMap在1.8之后是数组+链表/红黑树之间转换，hashtable没有这样转换机制）
  • 初始容量大小和扩容时候增加的大小不一样。

    —|HashMap|Hashtable
    -|-|-
    创建时不指定容量|16|11
    之后每次扩充|原来的2倍|原来的2n+1
    创建时指定容量|将其扩充为2的幂|直接使用给定容量

HashMap和TreeMap：
相比于HashMap来说， TreeMap 主要多了对集合中的元素根据键排序的能力以及对集合内元素的搜索的能力。

TreeMap它还实现了NavigableMap接口,提供了定向搜索、子集操作、逆序视图、边界操作这些方法，是基于红黑树实现。

HashMap的长度为什么是2幂次方
首先Hash值的范围很大，不能直接使用，要对数组的长度取模运算，得到的余数才是在数组中的下标。这个数组下标计算方法为：(n-1) & hash, 其中n是数组长度，hash是对应元素的hash值。当数组长度为2的幂次方后，n-1的二进制其实是一串全为1的数，在和hash进行与运算后就可以直接得到其下标，在计算机内部很快。

比如:

假设数组长度 n = 16（即 2^4），那么 n - 1 = 15，二进制表示为 0000 1111。

假设hash值为hash=29，二进制表示：hash = 0001 1101（即29）

计算 (n - 1) & hash：

0000 1111（15）

0001 1101（29）

位运算 AND 的结果是：0000 1101（即13）因此，hash = 29 对应的数组下标是13，这和29对16直接取模值是一样的。

ConcurrentHashMap 和 Hashtable 的区别

  • 底层数据结构：JDK1.7 的 ConcurrentHashMap 底层采用 分段的数组+链表 实现，JDK1.8 采用的数据结构跟 HashMap1.8 的结构一样，数组+链表/红黑二叉树。Hashtable 和 JDK1.8 之前的 HashMap 的底层数据结构类似都是采用 数组+链表 的形式，数组是 HashMap 的主体，链表则是主要为了解决哈希冲突而存在的。
  • 实现线程安全的方式（重要）：在 JDK1.7 的时候，ConcurrentHashMap 对整个桶数组进行了分割分段(Segment，分段锁)，每一把锁只锁容器其中一部分数据，多线程访问容器里不同数据段的数据，就不会存在锁竞争，提高并发访问率。到了 JDK1.8 的时候，ConcurrentHashMap 已经摒弃了 Segment 的概念，而是直接用 Node 数组+链表+红黑树的数据结构来实现，并发控制使用 synchronized 和 CAS 来操作。（JDK1.6 以后 synchronized 锁做了很多优化） 整个看起来就像是优化过且线程安全的 HashMap，虽然在 JDK1.8 中还能看到 Segment 的数据结构，但是已经简化了属性，只是为了兼容旧版本

    Hashtable(同一把锁) :使用 synchronized 来保证线程安全，效率非常低下。当一个线程访问同步方法时，其他线程也访问同步方法，可能会进入阻塞或轮询状态，如使用 put 添加元素，另一个线程不能使用 put 添加元素，也不能使用 get，竞争会越来越激烈效率越低。

ConcurrentHashMap线程安全具体实现

  • 首先可以总结一下JDK1.7和JDK1.8的实现不同：
    • 线程安全实现方式
    • Hash碰撞解决方法：1.7使用拉链法，1.8采用拉链法和红黑树（即链表超过一定长度就转为红黑树）
    • 并发度：1.7并发度是segment个数（16）,1.8最大并发度是Node数组大小。

JDK1.8之前：

由Segment数组结构和HashEntry数组结构组成。首先将数据分为一段一段（这个“段”就是Segment）的存储，然后给每一段数据配一把锁，当一个线程占用锁访问其中一个段数据时，其他段的数据也能被其他线程访问。

segment继承了ReentrantLock，所以这是一把可重入锁。HashEntrt用于存储键值对数据。

Segment 的个数一旦初始化就不能改变。 Segment 数组的大小默认是 16，也就是说默认可以同时支持 16 个线程并发写。

JDK1.8之后:

取消了Segment分段锁，采用Node + CAS + synchronized来保证并发安全。数据结构跟 HashMap 1.8 的结构类似，数组+链表/红黑二叉树。Java 8 中，锁粒度更细，synchronized 只锁定当前链表或红黑二叉树的首节点，这样只要hash不冲突，就不会产生并发，就不会影响其他 Node 的读写，效率大幅提升。

synchronized

这里涉及到一个优化的 synchronized，暂时还没记，先po上来gpt说的：

优化前（JDK 1.6之前）

  • 传统重量级锁

实现机制：传统的synchronized块由操作系统的监视器锁（monitor）实现，这种锁是重量级的，因为它依赖于操作系统的互斥锁（mutex）机制。
加锁和解锁：每次进入和退出synchronized块时，都会调用操作系统的原语进行加锁和解锁操作。

线程切换：当一个线程获取锁失败时，会被阻塞并进入等待队列，等待CPU资源重新调度，涉及到用户态和内核态的切换，开销大。

性能开销：高频率的加锁和解锁操作导致大量的上下文切换和系统调用，性能损失显著。

优化后（JDK 1.6及以后）

  • 偏向锁

优化机制：偏向锁假设大多数情况下锁是由同一个线程多次获得的。它在对象头中记录持有锁的线程ID，减少无竞争情况下的同步开销。

性能提升：如果偏向锁被同一个线程多次获得，无需CAS操作，大大减少了加锁和解锁的开销。

对比：

优化前：每次进入和退出synchronized块都需要操作系统调用。

优化后：无竞争情况下，线程只需检查和设置对象头中的线程ID，开销非常小。

  • 轻量级锁

优化机制：使用CAS操作在当前线程的栈帧中创建一个锁记录，将对象头中的锁指向这个锁记录。如果CAS操作成功，当前线程获得轻量级锁；如果失败，则自旋等待。

性能提升：减少了线程阻塞和上下文切换的开销。

对比：

优化前：线程竞争锁时立即进入阻塞状态，等待CPU重新调度。

优化后：线程竞争锁时首先自旋等待，避免了过早的阻塞和上下文切换。

  • 自适应自旋锁
    优化机制：自适应自旋锁根据前一次在同一锁上的自旋时间和锁的拥有者状态来决定自旋时间。

性能提升：通过动态调整自旋次数，减少不必要的自旋等待或过早阻塞，提高并发性能。

对比：

优化前：没有自旋等待机制，线程竞争失败立即阻塞。

优化后：通过自旋等待机制减少阻塞，提高锁的获取效率。

  • 锁消除

优化机制：通过逃逸分析技术，消除不必要的锁操作。如果对象没有逃逸到其他线程，则可以消除对该对象的同步操作。

性能提升：减少了无竞争情况下的同步开销。

对比：

优化前：即使锁对象只在单个线程内使用，仍然会进行同步操作。

优化后：在单线程情况下，JIT编译器可以优化掉不必要的同步操作。

  • 锁粗化

优化机制：将多个连续的锁操作合并成一个更大的锁操作，减少锁的获取和释放次数。

性能提升：减少频繁加锁和解锁的开销，提高代码的执行效率。

对比：

优化前：频繁进行加锁和解锁操作，导致性能开销大。

优化后：通过锁粗化减少锁操作的次数，降低了同步开销。

  • 锁升级和降级

优化机制：根据竞争情况，锁可以在偏向锁、轻量级锁和重量级锁之间动态升级和降级。

性能提升：适应不同竞争场景，尽量使用低开销的锁，只有在竞争激烈时才升级为重量级锁。

对比：

优化前：锁没有升级和降级机制，所有锁操作都使用重量级锁。

优化后：锁根据竞争情况动态调整，减少了不必要的重量级锁开销。

总结

优化前：

  • 依赖操作系统的监视器锁，使用重量级锁。
  • 每次加锁和解锁都需要系统调用，开销大。
  • 线程竞争失败立即阻塞，频繁的上下文切换导致性能损失。

优化后：

  • 偏向锁、轻量级锁、自适应自旋锁减少了无竞争或低竞争情况下的同步开销。
  • 锁消除和锁粗化减少了不必要的锁操作和频繁的加锁解锁次数。
  • 锁的动态升级和降级机制适应不同的竞争场景，提高了整体并发性能。
