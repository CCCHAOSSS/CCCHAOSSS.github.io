桶排序

桶排序从整体来看也是一个非比较排序，是计数排序的升级版，其主要是先把序列中的数据划分为一个个的范围，每个范围相当一个桶，将数据放入桶中，每个桶分别进行排序，然后取出拼接即可。计数排序不适用于序列中max-min范围较广或者序列中的数据不是整数的情况，但是桶排序没有这些限制，其主要步骤是将数据放入桶中，桶中的数据自己排序。

3. 在查找店铺功能中，如何使用Redis解决缓存穿透问题

首先，缓存三大问题：缓存穿透、缓存击穿、缓存雪崩。缓存穿透是指一直查询了（同一个？）缓存和数据库中都不存在的记录，因为在数据库中不存在，所以每次查询都不会有记录写入缓存，也就导致每次查询都会落到数据库上，给数据库带来压力。

要解决这个问题一般两个方法：缓存空值和布隆过滤器。

先说一下我用的是缓存空值。在查询一条记录时候，先看redis中有没有，没有再去查数据库。然后判断数据库查询的返回结果是否为null。如果是有记录的就把对应记录写入redis。如果是null，则把这条查询的条件作为key存入redis，对应value为“ ”（空字符串），下次再来的时候会命中redis这个记录，所以这里多一个判断命中的内容是否是空值的判断，是则返回null。

这里判断是否存在用isNotBlank，这个函数只有当真正有内容才会返回true（然后返回redis中命中的内容），空字符串和null都会返回false

再说一下布隆过滤器原理，布隆过滤器一般由一个数组和哈希函数组成。当要存一个东西时候，先用哈希函数进行运算，得到一个索引，将布隆过滤器中这些索引的位置设为1。然后当下一次查询过来时候可以先在用布隆过滤器检查它是否存在，根据得到的索引，看它在布隆过滤器中是否都为1。如果不是全为1那一定不存在，全为1再去查数据库等，虽然全为1也不一定会在数据库中存在，但是这是允许少量的错误的。布隆过滤器可以显著提升查询效率和节省存储空间。

4. 说一下互斥锁解决缓存击穿

这是基于redis的SETNX 互斥锁实现的。往redis中存入一个键值对，如果key不存在则成功存入，否则失败。

缓存击穿是对于一个热点数据，其缓存过期（失效后）大量的请求会打到数据库中，造成数据库宕机等。

缓存击穿是指当缓存中的热点数据过期或意外失效时，大量并发请求同时访问该数据，因为缓存未命中，所有这些请求都会直接打到数据库中去查询数据，导致数据库压力过大，甚至崩溃。

用互斥锁就是在对并发请求，若查询redis未命中，就要去申请该查询对象的锁，获得后才能继续进行数据库查询，并且查完后会进行缓存重建，然后释放锁。

5. 在优惠券秒杀业务中，如何使用Redis和Lua脚本解决超卖问题？

在优惠券秒杀业务中，使用 Redis + Lua脚本实现对用户秒杀资格的预检，同时用乐观锁解决秒杀产生的超卖问题。集群模式下， 基于Redis分布式锁解决一人一单的线程安全问题。

秒杀业务：多线程下优惠券库存要正确、下单一人只能一单、高并发下要高可用。

  • 优惠卷库存正确扣减：（最开始）使用乐观锁CAS版本号思想，因为在秒杀优惠卷时候，需要先判断库存够不够，足够才下单，这里的两个动作是非原子的，会有线程安全问题。在每个线程进行优惠卷扣减时候要判断此时的优惠卷是否还够，用Lua脚本保证这两个动作是原子的。Lua脚本中的动作：判断库存是否充足、然后条件都满足就下单（扣减库存并将订单信息存入redis）

  • 在上述基础上，（集群模式下）一人一单问题，一开始是使用synchronized关键字对用户加锁，先获取锁再进入createVoucherOrder方法（生成优惠卷订单方法）完全结束后再释放锁，此时一定提交了任务，单体模式下可以的。但是集群模式下就不行了，因为synchronized是JVM级别的锁，集群模式下有多个JVM，因此要用分布式锁。并且为了避免锁因为业务超时等问题被其他线程（同一个用户的其他线程）释放掉，在获取锁时候要将当前线程id存入锁中，在释放时候看当前是否与指定标识相同，相同才释放。

    • 这里用redisson提供的可重入锁。setnx不可重入。
